# Paradise Timer チャット機能仕様書

## 1. 概要

Paradise Timer に、ユーザー同士で利用できるルーム型チャット機能を追加する。

- 既存のタイマー機能は `/` で引き続き提供する。
- チャット機能は `/chat/` 以下で提供する。
- ユーザーは複数のルームを作成できる。
- ルームへの参加にはオーナーの承認が必要となる。
- データは Deno KV に保存する。
- 既存のタイマー画面右上に `/chat/` へ移動する「チャット」ボタンを追加する。

## 2. 対象範囲

### 2.1 初期リリースに含む機能

- メールアドレスとワンタイムパスワードによるログイン
- ユーザープロフィールと表示名
- 複数ルームの作成と管理
- URLを使ったルーム参加申請
- オーナーによる申請の承認・拒否
- 閲覧のみ／書き込み可の権限管理
- テキストメッセージのリアルタイム送受信
- 未読件数と参加申請通知
- 参加申請のメール通知
- 投稿削除
- ルーム削除
- アカウント削除

### 2.2 初期リリースに含めない機能

- 画像、動画、ファイルなどの添付
- メッセージ編集
- 既読者一覧
- 入力中表示
- 音声通話、ビデオ通話
- リアクション、スタンプ
- スレッド、返信引用
- ルーム所有権の譲渡

## 3. URLと画面構成

| URL                            | 内容                                   |
| ------------------------------ | -------------------------------------- |
| `/chat/`                       | チャットダッシュボード                 |
| `/chat/login`                  | メールアドレス入力、OTP入力            |
| `/chat/profile`                | 表示名、メール通知設定、アカウント削除 |
| `/chat/rooms/new`              | ルーム作成                             |
| `/chat/rooms/{roomId}`         | チャット、参加申請、承認待ち表示       |
| `/chat/rooms/{roomId}/members` | 参加申請、メンバー、権限管理           |

未ログイン状態でルームURLへアクセスした場合、ログイン完了後に元のルームURLへ戻す。

## 4. 認証

### 4.1 認証方式

メールアドレスと6桁のワンタイムパスワード（OTP）を使用する。

1. ユーザーがメールアドレスを入力する。
2. サーバーが6桁のOTPを発行する。
3. Resendを使ってOTPをメール送信する。
4. ユーザーがOTPを入力する。
5. 検証成功後、サーバーがセッションCookieを発行する。
6. 初回ログイン時は表示名設定へ進む。
7. 通常ログイン時はアクセス元の画面へ戻る。

### 4.2 OTP仕様

- 6桁の数字とする。
- 有効期限は10分とする。
- 再送は前回送信から60秒後に可能とする。
- 1つのOTPに対する入力失敗は5回までとする。
- OTPは平文保存せず、`AUTH_SECRET`を使ってハッシュ化して保存する。
- 新しいOTPを発行した時点で、同じメールアドレスの古いOTPを無効化する。
- 認証要求への応答から、アカウントの存在有無を推測できないようにする。

### 4.3 セッション仕様

- セッション有効期間は30日とする。
- セッションIDは十分な長さの暗号学的乱数とする。
- Cookieには `HttpOnly`、`Secure`、`SameSite=Lax` を設定する。
- ログアウト時にサーバー側セッションを削除する。
- アカウント削除時に、そのユーザーの全セッションを削除する。

### 4.4 メール送信

Resendを利用する。Deno Deployには次の環境変数を設定する。

```text
RESEND_API_KEY
AUTH_SECRET
EMAIL_FROM
CHAT_PUBLIC_ORIGIN
```

本番送信ではResendに送信元ドメインを設定する。 `CHAT_PUBLIC_ORIGIN`
には、参加申請メールの管理リンクに使う公開HTTPSオリジンを
設定する。パス、クエリ、フラグメントは含めない。

## 5. ユーザープロフィール

- ユーザーは内部の不変なユーザーIDで識別する。
- メールアドレスはログイン識別子として利用し、他ユーザーには表示しない。
- 初回ログイン時に表示名を設定する。
- 表示名は1〜30文字とする。
- 表示名は後から変更できる。
- 表示名の重複を許可する。
- 表示名を変更した場合、過去の投稿にも新しい表示名を表示する。
- 表示名変更はルーム所有権、参加状態、権限に影響しない。
- HTMLとして解釈せず、常にテキストとして表示する。

## 6. ルーム

### 6.1 ルーム作成

- 認証済みユーザーは複数のルームを作成できる。
- 作成者をルームオーナーとする。
- ルーム名は必須で、1〜50文字とする。
- 説明は任意で、最大200文字とする。
- ルームIDには推測困難なランダムIDを使用する。
- ルームURLは `/chat/rooms/{roomId}` とする。

### 6.2 ルーム管理

- オーナーはルーム名と説明を変更できる。
- オーナーはルームを削除できる。
- 初期リリースでは所有権を譲渡できない。
- オーナーは自分自身をルームから削除できない。
- オーナーは常に全メッセージを閲覧・投稿できる。

### 6.3 ルーム削除

- 削除確認時にルーム名の再入力を要求する。
- 削除時にルーム、メッセージ、参加申請、メンバー、既読状態をすべて削除する。
- 削除後の復元機能は初期リリースでは提供しない。

## 7. 参加申請

### 7.1 参加フロー

1. ユーザーがルームURLへアクセスする。
2. 未ログインの場合はログインする。
3. 未参加の場合は参加申請画面を表示する。
4. ユーザーが参加申請を送信する。
5. 申請者には「承認待ち」を表示する。
6. オーナーへ画面内通知とメール通知を送る。
7. オーナーが「閲覧のみ」または「書き込み可」を指定して承認するか、申請を拒否する。
8. 承認された時刻を、そのメンバーの閲覧開始時刻として保存する。

### 7.2 申請状態

- `pending`: 承認待ち
- `approved`: 承認済み
- `rejected`: 拒否
- `removed`: 参加解除

### 7.3 再申請

- 拒否されたユーザーは24時間後に再申請できる。
- 参加解除されたユーザーは再申請できる。
- 再承認された場合、閲覧開始時刻を再承認時刻に更新する。
- 再承認前の投稿は閲覧できない。

## 8. 権限

| 役割／状態     | 閲覧 | 投稿 | 自分の投稿削除 | 全投稿削除 | 申請管理 | 権限変更 | ルーム管理 |
| -------------- | ---: | ---: | -------------: | ---------: | -------: | -------: | ---------: |
| オーナー       |    ○ |    ○ |              ○ |          ○ |        ○ |        ○ |          ○ |
| 書き込み可     |    ○ |    ○ |              ○ |          × |        × |        × |          × |
| 閲覧のみ       |    ○ |    × |              × |          × |        × |        × |          × |
| 承認待ち       |    × |    × |              × |          × |        × |        × |          × |
| 拒否／参加解除 |    × |    × |              × |          × |        × |        × |          × |

- オーナーは承認時に「閲覧のみ」または「書き込み可」を選択する。
- オーナーは承認後も権限を変更できる。
- 権限変更は即時反映する。
- 「閲覧のみ」から「書き込み可」へ変更しても、閲覧開始時刻は変更しない。
- 「書き込み可」から「閲覧のみ」へ変更した場合、それ以降の投稿を禁止する。
- すべての権限は画面表示だけでなく、サーバーAPI側で検証する。

## 9. メッセージ

### 9.1 投稿

- 初期リリースはテキストのみ対応する。
- 本文は1〜2,000文字とする。
- Enterで送信し、Shift+Enterで改行する。
- 空白だけの投稿は拒否する。
- HTMLとして解釈せず、常にテキストとして表示する。
- サーバーで投稿権限とレート制限を検証する。
- メッセージには投稿者ID、ルームID、本文、投稿時刻、削除状態を保存する。

### 9.2 閲覧範囲

- オーナーはルーム作成時以降の全投稿を閲覧できる。
- 承認済みメンバーは `message.createdAt >= member.visibleFrom`
  の投稿だけ閲覧できる。
- 承認前、拒否中、参加解除中はメッセージを取得できない。
- サーバーAPIが閲覧開始時刻による絞り込みを行い、クライアントだけに依存しない。

### 9.3 履歴

- 最新50件を最初に取得する。
- 上方向への追加読込で、それ以前の50件を取得する。
- 閲覧開始時刻より前の投稿は、ページング時にも返さない。
- メッセージは期限を設けず保存する。

### 9.4 編集と削除

- 投稿後の編集はできない。
- 投稿者は自分の投稿を削除できる。
- オーナーはルーム内のすべての投稿を削除できる。
- 閲覧のみユーザーは投稿を削除できない。
- 削除時は本文を削除し、画面には「削除されたメッセージ」と表示する。
- 削除された本文は復元できない。

## 10. リアルタイム更新

- メッセージ受信と通知更新にはServer-Sent Events（SSE）を使用する。
- メッセージ送信、申請、承認、権限変更は通常のHTTP APIを使用する。
- SSE切断時はブラウザの自動再接続を利用する。
- 再接続時は最後に受信したイベントID以降を取得し、取りこぼしを補完する。
- Deno KVを正本とし、プロセス内メモリだけに依存しない。
- 権限変更または参加解除後は、対象ユーザーの接続でも以後のイベントを送信しない。

## 11. 通知と未読

### 11.1 画面内通知

- `/chat/` にルームごとの未読件数を表示する。
- オーナーには参加申請数のバッジを表示する。
- 新しい申請、承認、拒否、権限変更、参加解除を画面内で通知する。

### 11.2 メール通知

- 参加申請時にオーナーへメールを送る。
- 同一申請について重複メールを送らない。
- ユーザーはプロフィール設定でメール通知をオフにできる。
- 初期値はメール通知オンとする。
- メール本文にはルーム名、申請者の表示名、管理画面へのリンクを含める。

### 11.3 未読管理

- ルームごと、ユーザーごとに最終閲覧位置を保存する。
- 自分が投稿したメッセージは未読に含めない。
- ルームを表示して最新位置まで到達した時点で既読位置を更新する。

## 12. アカウント削除

- プロフィール画面からアカウントを削除できる。
- 所有中のルームがある場合は、先にルームを削除しないと退会できない。
- アカウント削除時にメールアドレス、OTP、セッション、通知設定などの個人情報を削除する。
- 過去の投稿は残す。
- 退会ユーザーの投稿者名は「退会したユーザー」と表示する。
- 退会後のログインは新規ユーザーとして扱う。

## 13. 利用上限とレート制限

| 対象                        |                         上限 |
| --------------------------- | ---------------------------: |
| 1ユーザーが所有できるルーム |                         20個 |
| 1ルームのメンバー           |                        100人 |
| メッセージ送信              |                 10秒間に10件 |
| 参加申請                    |       1ユーザーにつき1日10件 |
| OTP送信                     | メールアドレスごとに1時間5回 |
| OTP入力失敗                 |             1コードにつき5回 |
| メッセージ本文              |                    2,000文字 |
| ルーム名                    |                       50文字 |
| 表示名                      |                       30文字 |
| ルーム説明                  |                      200文字 |

- OTP送信はIPアドレス単位でも制限する。
- 上限到達時は処理を拒否し、理由と再試行可能時刻を返す。
- APIは適切なHTTPステータスコードを返す。

## 14. セキュリティ要件

- 認証、ルーム所属、権限、閲覧開始時刻をすべてサーバー側で検証する。
- メールアドレス、セッションID、OTP、APIキーをクライアントへ公開しない。
- ユーザー入力をHTMLとして挿入しない。
- 状態変更APIにはCSRF対策を行う。
- セッション固定攻撃を防ぐため、ログイン成功時に新しいセッションを発行する。
- OTP、セッション、申請、投稿APIにレート制限を設ける。
- エラーメッセージからメールアドレスの登録状況を推測できないようにする。
- Resend APIキーと認証秘密鍵はDeno Deployのシークレット環境変数で管理する。
- ログへOTP、セッションID、メール本文、メールアドレス全体を出力しない。

## 15. Deno KVデータモデル案

実装時のキー名前空間は、既存機能やPerfpadと衝突しないよう、先頭を `chat`
に統一する。

```text
["chat", "users", userId]
["chat", "usersByEmail", normalizedEmail]
["chat", "otp", normalizedEmail]
["chat", "sessions", sessionId]
["chat", "rooms", roomId]
["chat", "roomsByOwner", ownerId, roomId]
["chat", "members", roomId, userId]
["chat", "requests", roomId, userId]
["chat", "messages", roomId, messageId]
["chat", "messageEvents", roomId, messageId]
["chat", "readPositions", roomId, userId]
["chat", "notifications", userId, notificationId]
["chat", "rateLimits", category, subject, window]
["chat", "roomsByMember", userId, roomId]
["chat", "events", eventId]
["chat", "eventSequence"]
```

メッセージIDには時系列順に並べられるIDを使用する。メールアドレスは小文字化などの正規化を行い、検索用キーとして利用する。

### 16.1 認証・プロフィール


## 17. 主要な受け入れ条件

1. メールOTPでログインし、初回に表示名を設定できる。
2. 表示名を変更すると、過去の投稿にも新しい名前が表示される。
3. ユーザーが複数のルームを作成できる。
4. ルームURLから参加申請できる。
5. 承認前のユーザーはメッセージを閲覧・投稿できない。
6. オーナーが申請を閲覧のみ／書き込み可で承認できる。
7. 承認されたユーザーには承認時刻以降の投稿だけが返る。
8. オーナーが承認後に権限を変更できる。
9. 閲覧のみユーザーはAPIを直接呼んでも投稿できない。
10. 新着メッセージと参加申請がリアルタイムに反映される。
11. オーナーに参加申請の画面内通知とメール通知が届く。
12. 投稿者が自分の投稿を削除でき、オーナーが全投稿を削除できる。
13. 参加解除・拒否・ルーム削除・アカウント削除が仕様どおり機能する。
14. すべての利用上限とレート制限がサーバー側で適用される。
15. 既存のタイマー機能が `/` で従来どおり動作する。
