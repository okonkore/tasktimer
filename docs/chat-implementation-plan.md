# Paradise Timer チャット機能 実装計画

## 1. この計画の使い方

チャット機能を、単独で検証・コミットできる単位へ分割して順番に実装する。

- 原則として1タスクにつき1コミットとする。
- 各タスクでは、そのタスクの範囲外の機能を先回りして実装しない。
- 各タスク開始時に `docs/chat-specification.md` と本書を読む。
- 各タスクの完了条件を満たし、テストを通してからコミットする。
- 次のタスクは、直前のタスクがコミット済みの状態から開始する。
- 認証、権限、閲覧範囲、削除、リアルタイム配信はセキュリティ境界を含むため、GPT-5.6 Solを使用する。
- 画面、フォーム、通常CRUD、表示ロジックはGPT-5.6 Terraを使用する。

モデル選択の基本方針:

| モデル | 用途 |
|---|---|
| GPT-5.6 Sol / high | 認証、認可、データ整合性、リアルタイム、セキュリティレビュー |
| GPT-5.6 Terra / high | APIとUIをまたぐ通常実装、状態管理、複数画面の結合 |
| GPT-5.6 Terra / medium | 独立したUI、スタイル、単純なCRUD、文書更新 |

## 2. 実装前の準備

### 2.1 外部サービス

実装開始前またはOTPメール送信タスクまでに、次を準備する。

- Resendアカウント
- 本番送信用に検証済みの送信元ドメイン
- Resend APIキー
- 送信元メールアドレス

### 2.2 Deno Deploy環境変数

```text
RESEND_API_KEY
AUTH_SECRET
EMAIL_FROM
```

`AUTH_SECRET`には十分な長さの暗号学的乱数を設定する。秘密値はGitへコミットしない。

### 2.3 現在の仕様コミット

仕様書コミット `cf1e0e0 Document chat feature specification` は、実装開始前に `origin/main` へプッシュする。

### Task 1: ルーティング基盤とテスト基盤

推奨モデル: **GPT-5.6 Terra / medium**

目的:

- 既存タイマーを壊さずに `/chat/` 以下を配信できるサーバールーティング基盤を作る。
- 後続タスクがAPI単体テストを追加できる構成にする。

実装範囲:

- `server.ts` のルーティング責務を分割する。
- `/` と既存タイマーAPIの互換性を維持する。
- `/chat/` 以下のHTML、JS、CSSを配信するための骨組みを追加する。
- `/api/chat/health` を追加する。
- Denoテストからハンドラを直接呼べる構成にする。
- テスト用のインメモリDeno KV利用方針を決める。

完了条件:

- `/` で既存タイマーが表示される。
- `/chat/` が仮のチャット画面を返す。
- `/api/chat/health` が200を返す。
- 既存のタイマー文書APIが動作する。
- テスト、型チェック、lintが通る。

コミット名:


### Task 3: OTP発行、メール送信、検証API

推奨モデル: **GPT-5.6 Sol / high**

目的:

- メールアドレスと6桁OTPによる安全な認証の前半を実装する。

実装範囲:

- `POST /api/chat/auth/request-otp` を追加する。
- `POST /api/chat/auth/verify-otp` を追加する。
- 6桁OTPを暗号学的乱数で生成する。
- `AUTH_SECRET`を使いOTPをハッシュ化して保存する。
- 10分の有効期限、60秒の再送待機、5回の試行上限を実装する。
- Resendによるメール送信を実装する。
- テスト時にメール送信を差し替えられる設計にする。
- アカウント存在有無を推測できない応答にする。

完了条件:

- 正しいOTPで認証できる。
- 誤ったOTP、期限切れ、試行上限超過を拒否する。
- OTPの平文がKVとログに残らない。
- 再送時に古いOTPが無効になる。
- Resend失敗時に安全なエラー処理を行う。
- 認証APIテストが通る。

コミット名:

```text
Add email OTP authentication
```

完了条件:

- Cookieなしでは保護APIを利用できない。
- 有効なCookieで自分のユーザー情報を取得できる。
- ログアウト後のセッションを再利用できない。
- 期限切れセッションを拒否する。

- Terra / mediumで始めたタスクでも、認証・認可・データ削除へ変更範囲が広がった場合は、作業を止めてSol / highへ切り替える。
- Sol / highを使うタスクでは、実装だけでなく攻撃者視点の否定テストを必須とする。
- 1タスクの差分が大きくなりすぎた場合は、コミット可能な下位タスクへ分割する。
