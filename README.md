# Paradise Timer

タスクストックとタイムラインを使うタイマーです。タイマーは `/`、保存文書APIは
`/api/state` と `/api/documents`、ユーザー間チャットは `/chat/` で提供します。
タイマーとチャットは同じDeno KVを利用しますが、キーの先頭をそれぞれ `tasktimer`
と `chat` に分けています。共有先のPerfpadでもこの2つの接頭辞を
使わないでください。

チャットの機能仕様は
[docs/chat-specification.md](docs/chat-specification.md)、実装単位と検証範囲は
[docs/chat-implementation-plan.md](docs/chat-implementation-plan.md)、受け入れ条件と
テストの対応は
[docs/chat-acceptance-checklist.md](docs/chat-acceptance-checklist.md)
にあります。

## ローカルでの検証

Deno 2系を利用します。

```sh
deno task fmt:check
deno task check
deno task test
deno task lint
node --check chat/client.js
git diff --check
```

実送信を伴うローカル起動には、次節の4環境変数を設定してから実行します。

```sh
deno task dev
```

## Deno Deploy

`deno.json` のエントリーポイントは `server.ts` です。Deno Deployのプロジェクトに
Deno KVデータベースをデフォルトDBとしてアタッチし、次の値をSecretまたは環境変数
として設定します。

| 変数                 |     必須 | 内容                                                                                       |
| -------------------- | -------: | ------------------------------------------------------------------------------------------ |
| `AUTH_SECRET`        |     必須 | UTF-8で32バイト以上のランダムな秘密値。OTPのハッシュに使用                                 |
| `RESEND_API_KEY`     |     必須 | Resendの送信権限を持つAPIキー                                                              |
| `EMAIL_FROM`         |     必須 | Resendで検証済みドメインの送信元アドレス                                                   |
| `CHAT_PUBLIC_ORIGIN` | 本番必須 | `https://timer.example.com` のような公開オリジン。パス、クエリ、末尾のスラッシュは付けない |

`AUTH_SECRET`、APIキー、メールアドレスをリポジトリへ保存しないでください。
`CHAT_PUBLIC_ORIGIN` が未設定または不正な場合、チャットは動作しますが参加申請
メールは送信されません。

Resend側では送信ドメインのDNS検証を完了し、そのドメインのアドレスを `EMAIL_FROM`
に設定します。OTPメールの失敗はアドレス列挙を防ぐためAPI上は成功と
同じ応答になります。また、参加申請メールの失敗は申請自体を失敗させません。
したがってデプロイ後はResendの管理画面でも実配送を確認してください。

デプロイ前に次を記録します。

- デプロイ対象コミットと直前の正常なデプロイ
- 4環境変数の設定有無（秘密値そのものを作業ログへ書かない）
- アタッチしたKVデータベース
- 可能であればKVのエクスポートまたはバックアップ

## デプロイ後の確認

`GET /api/chat/health` が200と `{"ok":true,"service":"chat"}`
を返すことを確認します。このエンドポイントは
liveness確認であり、KV接続、環境変数、Resend配送までは検査しません。

続けて、次の順でスモークテストします。

1. `/` でタイマーを開き、タスクストックとタイムラインを保存する。
2. 「新規作成」「保存」「開く」で文書全体が復元されることを確認する。
3. `/chat/` を開き、OTPの実メールでログインして初回表示名を設定する。
4. オーナー、閲覧のみ、書き込み可の3アカウントを別ブラウザプロファイルで用意する。
5. ルームURLから2人が申請し、画面内通知と参加申請メールを確認する。
6. 2種類の権限で承認し、承認前の投稿が見えないことを確認する。
7. 閲覧のみユーザーの投稿が拒否され、書き込み可ユーザーの投稿がSSEで即時表示されることを確認する。
8. 未読、権限変更、参加解除、再申請、投稿削除を確認する。
9. ブラウザを切断して50件を超えて投稿した後に再接続し、上方向の履歴読込に欠落がないことを確認する。
10. 確認用ルームを名前再入力で削除し、所有ルームのない確認用アカウントを削除する。

HTMLは再検証されるよう `no-cache`、JSとCSSは最大1時間キャッシュされます。
デプロイ確認時は、必要に応じてブラウザを再読み込みしてください。

## ロールバック

障害時はDeno Deployで直前の正常なデプロイへ戻すか、そのコミットを再デプロイ
します。設定変更が原因なら、先に環境変数を直前の値へ戻します。その後、上記の
health、タイマー文書、OTP、ルーム閲覧と投稿の確認を繰り返します。

コードのロールバックはKVの内容を巻き戻しません。現在の実装に破壊的な
スキーマ移行はありませんが、投稿本文、ルーム、アカウントの削除は復元できません。
データ復元が必要な場合は、コードを戻す前に取得済みのKVバックアップから別DBへ
復元し、内容を確認してから切り替えます。

`AUTH_SECRET` を変更すると発行済みOTPは検証できなくなり、OTPレート制限の
識別値も変わります。既存セッションのハッシュには使っていませんが、緊急時以外は
デプロイと同時に不用意にローテーションしないでください。

## 運用上の上限と注意

- 1ユーザーの所有ルームは20、1ルームはオーナーを含め100人です。
- OTPはメールごと5回/時、IPごと20回/時、再送60秒、入力失敗5回、有効10分です。
- 参加申請は1ユーザー10回/日、拒否後の再申請は24時間後です。
- 投稿は1ユーザー10件/10秒、本文は2,000文字、履歴は50件ずつ取得します。
- SSEは接続ごとにKVを約1秒間隔で確認し、約15秒間隔でheartbeatを送ります。
- SSEイベントに一律の保存期限はありません。ルームまたは関連アカウントの削除時に削除されます。
- 未読集計はルーム履歴、削除は関連する通知やイベントを走査します。非常に大きなルームではKV読取量とリクエスト時間を監視してください。
- 参加申請メールは重複防止を優先し、送信失敗した同一申請を自動再送しません。
- タイマーの保存APIは従来どおり認証なしです。公開範囲を限定できない環境では、機密情報を保存しないでください。
- 添付、編集、リアクション、スレッド、通話、所有権譲渡は初期リリース対象外です。

自動テストはOTP、セッション、プロフィール、複数ルーム、申請と拒否、2権限、
閲覧開始時刻、メッセージと削除、SSE再送、未読、メール通知、レート制限、
ルーム・アカウント削除、およびタイマー文書の回帰を検証します。ブラウザ固有の
Enter/Shift+Enter、スクロール位置、複数タブの表示は上記スモークテストでも確認します。
